# クリティカルパスと高度な分析

すべてのスケジュールには、プロジェクトの終了時期を左右する、タスクの最長のチェーンがあります。これがクリティカルパスです。それ以外のすべてにはフロートがあります — 終了日に触れずに遅れられる余地です。このガイドでは「どのバーが赤いか」の先に踏み込み、トータルフロート／フリーフロート／干渉フロート、準クリティカルな作業、複数の等しくクリティカルなパス、Hammock、ハードピンとその上流への影響、プロジェクト間の外部リンクを扱います。

## ここで学ぶこと

- クリティカルパスの読み取り方と、トータルフロート・フリーフロート・干渉フロートの違い。
- 準クリティカルな作業：しきい値の設定とアンバー表示の見分け方。
- 複数のクリティカルパスが同時に存在するケース：いつ起こり、どう確認するか。
- ハードピンとフロートへの影響。上流で生じる負のフロートを含む。
- Hammock（Level of Effort）：それが何をし、何をしないか。
- プロジェクト間の外部リンク：凍結されたアンカー、更新、「ソースが読み込まれていません」ステータス。
- コンテキストメニューやリボンからのパスのトレース。
- プロジェクト設定の **計算** セクション。

[Nieuwbouw Appartementencomplex De Vaart](examples://showcase-appartementencomplex.ifc) を見ながら進めてください — 3棟の塔が並行して進む、大規模で「なんでも詰まった」ショーケースで、このガイドのほぼすべてのトピックが確認できます。複数のクリティカルパス、準クリティカルな作業、Hammock、ハードピン、そして別ファイルへの外部リンクです。

## クリティカルパスを読み取る

**F5**（または **計算** ボタン）を押してスケジュールを実行します。画面下部のステータスバーには、例えば「クリティカルパス：N タスク、M 稼働日」と表示されます — クリティカルパス上のタスク数と合計期間です。ガントチャートでは、クリティカルタスクには専用の（赤い）バーの色が付きます。フロートがなく、1日の遅れがそのままプロジェクトの終了日を押し出すタスクです。

タスクをダブルクリックし、**CPM 結果** セクションで正確な数値を確認します。**最早開始**、**最早終了**、**最遅開始**、**最遅終了**、**トータルフロート**、**フリーフロート**、（該当する場合は）**干渉フロート**、そして **クリティカルパス** 上にあるかどうかです。これらのフィールドをタスクテーブルの列としても見たいですか？ **表示 → 列…** でチェックを入れてください。

### トータルフロート、フリーフロート、干渉フロート

- **トータルフロート** — プロジェクトの終了日に触れずに、タスク全体としてどれだけ遅れられるか。ゼロならクリティカルです。
- **フリーフロート** — 直後の後続タスクに触れずに、どれだけ遅れられるか。トータルフロートより小さいことがあります。あるタスクがトータルフロートを持っていても、1日遅れただけで直後の後続タスクがすでに動いてしまう場合があります（その後続タスク自身は、終了日に触れないだけの十分なフロートを持っています）。
- **干渉フロート** — 両者の差（トータルフロート − フリーフロート）：終了日には触れないものの、後続タスクの「邪魔になる」フロートの部分です。ゼロならフリーフロートとトータルフロートが等しく、自分のフロート内で遅れても誰にも影響しません。

## 準クリティカルな作業

トータルフロートがわずかでもゼロでないタスクは脆弱です。ちょっとした遅れで結局クリティカルになってしまいます。**プロジェクト情報 → 計算 → 準クリティカルとしてマーク** でこれをオンにし、稼働日（または、期間の表示設定によっては時間）単位の **しきい値** を設定します。トータルフロートがゼロより大きく、そのしきい値以下のすべてのタスクは、ガント上でアンバー色のバーになります — クリティカルの赤と、十分なフロートがある緑の中間です。

大規模なショーケースではしきい値が3稼働日に設定されています。そのため **Tower C** の最終検査は、ちょうど3稼働日のトータルフロートを持ちます — しきい値ぎりぎり内側です — 一方、まったく同じ **Tower A** と **Tower B** の最終検査はフロートがゼロで、正真正銘のクリティカルです。Tower C は、1つのわずかに短い仕上げタスクを除いて、タスクと期間が他の2棟とまったく同じです。その小さな違いだけで、クリティカルから準クリティカルへと移るのに十分なのです。

## 複数のクリティカルパス

通常、最長のチェーンはちょうど1つですが、2つ以上のチェーンがまったく同じ長さになることもあります — その場合、両方（またはすべて）が等しくクリティカルになります。**複数フロートパス**（**プロジェクト情報 → 計算**）をオンにすると、これが計算されます。**方式**（**フリーフロート（ピーリング）** または **トータルフロート（ランキング）**）と **最大パス数** を選びます。すると、すべてのタスクに **フロートパス** の番号が付きます（1＝最もクリティカル）。フロートパスを持たないタスクは、計算されたどのパスにも含まれていません。

大規模なショーケースでは、Tower A と Tower B はタスクと期間が完全に対称です — まったく同じ時刻に終了します。**複数フロートパス** をオンにすると、結果に複数のパスが表示されます（計算結果の `criticalPaths.length` が1より大きくなります）。単一の最長チェーンではなく、プロジェクトを貫く複数の等しくクリティカルなチェーンです。これは「1つのクリティカルパスと、その脇にいくつかの準クリティカルな作業がある」とは異なる信号です — それらのパスの *いずれか* での遅れが、等しく終了日を直撃することを意味するため、1つのチェーンだけに注意を集中することはできません。

## ハードピンとフロートへの影響

**ハードピン**（MSO または MFO 制約上の **必須（ピン留めロジック）** チェックボックス）は、先行タスクが論理的に矛盾していても、タスクをその日付に固定します。大規模なショーケースでは、これを「Wegafzetting gemeente (vergunde stremmingsperiode)」（自治体の道路封鎖、許可された封鎖期間）に使っています。自治体はまさにその許可された日にしか封鎖を認めません、それだけです — ネットワークのロジックはそれに合わせて曲げられます。

上流への影響は、見通しにくい厄介な部分です。ピン留めされたタスクの先行タスクが、ピンの日付までに利用可能な時間より多くを必要とする場合、その先行タスクに **負のフロート** が現れます。したがって負のフロートは計算エラーではありません。エンジンが「この先行チェーンは、もはやピンが許す時間には収まりません」と伝える方法なのです。ハードピンの上流に負のフロートが見えたら、問うべきは「何が壊れているのか」ではなく「この2つのうちどちらを譲るべきか — ピンの日付か、その前のチェーンの期間か」です。

注：大規模なショーケースでは、「Wegafzetting gemeente」の周辺のチェーン全体 — ピン留めされたタスク自体を含む — は、（状態日よりずっと前に）実績開始・実績終了として、すでに完全に完了しています。そのため、ピンタスク自体を含むフェーズ1のチェーン全体に、わずかな残存の負のフロートが見られますが、これは上記で説明した「先行タスクが収まらない」シナリオではなく、すでに完了しているタスクと状態日が組み合わさったときの特徴です。このシナリオを純粋な形で確認するには、一時的に状態日をクリアして（**ベースラインと進捗** リボングループ、**状態日を解除** ボタン）再計算してください — ピンタスク自体はトータルフロートゼロに戻り、意図的に先行チェーンをピンの日付までの余地より長くして初めて負のフロートが現れます。

## Hammock（Level of Effort）

**Hammock**（プロパティパネルの **Hammock（派生期間）** チェックボックス）は、それ自体の期間の入力を持たないタスクです。その開始と終了は、自身の関係から自動的に導かれます。受け取る **FS**／**SS** 関係が **開始ドライバー**（最も早い開始）を、受け取る **FF**／**SF** 関係が **終了ドライバー**（最も遅い終了）を供給します — Hammock のチェックボックスをオンにすると、パネルには両方が読み取り専用で表示されるため、どのタスクがその期間を決めているかが正確にわかります。終了ドライバーがない場合、期間はゼロ長に戻り、パネルに警告が表示されます。

Hammock がすること：一種の包括的なバーとして、自分で期間を維持しなくても作業のひとかたまりの全期間を示します — 例えば、根底の作業とまったく同じ期間だけ続く「監督業務」や「現場の一般管理費」などに便利です。Hammock がしないこと：CPM の計算に影響するリソースやロジックを自分自身では一切持ちません — それは派生的なビューであり、駆動するタスクではありません。大規模なショーケースでは、これを「Ruwbouw toren A (LOE)」（躯体・構造、Tower A）に使っています。Tower A の最初の本当の躯体・構造タスクが始まると同時に始まり、最後のタスクが終わると同時に終わる Hammock で、その間のどこにも自分自身は存在しません。

## プロジェクト間の外部リンク

大規模なプロジェクトは、複数の別々に管理されるサブスケジュールから成ることがあります — 例えば自分自身のマスタースケジュールと、別の施工業者が管理する外構工事パッケージなどです。**外部リンク**（**関係** タブのボタンから開く **外部（プロジェクト間）リンク** ウィンドウ）は、そのファイルをドキュメントとして開くことなく、そのような別のファイル内のタスクへの関係を記録します。

最近使用したファイルから **ソースファイル** を選ぶか（読み取り専用で読み込まれ、ドキュメントとして開かれることはありません）、ソースファイルが手元にない場合はプロジェクトID、タスクID、アンカー日付を **手動** で入力します。それから **方向**（先行または後続）、**関係タイプ**（FS/SS/FF/SF）、**ラグ** を選びます。**アンカー日付** — リンクした瞬間のソースタスクの日付 — は自分のファイル内に凍結されます。ソースプロジェクトが変更されても、その日付は自動的には追従しません。

ソースファイルが更新されたかどうか知りたいですか？ **関係** タブの **外部リンク** セクションに移動し、（リンクごとに）**このリンクを更新** または（一括で）**外部アンカーを更新** をクリックして、ソースファイルを再読み込みしアンカーを更新します。ソースファイルが利用できない場合 — 移動、リネーム、あるいは同梱されていない場合 — リンクには「更新するには再インポートしてください」というツールチップとともに **古い** というラベルが表示されます。この場合、アプリは凍結されたアンカーがまだ有効かどうかを自分で検証できません。

大規模なショーケースでは、まさにこの最後のパターンをあえて示しています。タスク「Bestrating parkeerterrein」（駐車場の舗装）は、あえてサンプルに同梱されていない外構工事の下請業者のソースファイルにリンクされています。タスクを開くと、リンクは「古い」ステータスとともに一覧表示されます — 常に完璧に更新されるリンクではなく、外部のソースファイルが利用できなくなったときに何が起きるかを正直に示す実演です。

## パスをトレースする

あるタスクに上流・下流でどのタスクが影響しているかを正確に確認したいですか？ タスクを右クリックして **パスをトレース** を選びます（再度オフにするには **パスのトレースを停止**）— これにより、先行タスクと後続タスクのチェーン全体が一度にハイライトされます。より的を絞った作業には、リボン（**計画** または **関係** タブの **パストレース** リボングループ）に **先行タスク**／**後続タスク** という別のボタンのペアがあります。両方オフなら何も表示されず、片方オンならその方向だけが表示され、両方オンならコンテキストメニューのコマンドと同じになります。トレースはまた、論理的につながっているすべてのタスクと、実際にその日付を **駆動** しているタスク（関係テーブルに表示されるのと同じ「駆動」関係）とを区別します — つまり、何がつながっているかだけでなく、実際に何が舵を取っているかがわかります。

## 計算設定

**プロジェクト情報**（Backstage → プロジェクト情報、または **プロジェクト情報** ウィンドウ）内の **計算** セクションには、この特定のプロジェクトに属する計算オプションがまとめられています — これらはアプリではなくファイルに属するので、同じファイルを開いた同僚も同じ結果を得られます。

- **クリティカルの定義** — **トータルフロート ≤ しきい値**（既定のしきい値0）または **最長パス**。後者は、フロートの数値にかかわらず、ネットワークを貫く最長のチェーンに基づいてタスクをクリティカルとマークします。
- **フロート計算** — 開始側と終了側の両方を持つタスクについて、トータルフロートをどう決めるか：**最小（開始/終了）**（既定）、**開始フロート**、**終了フロート**。
- **オープンエンドタスクをクリティカルにする** — 後続タスクを持たないタスクを自動的にクリティカル扱いします。
- **準クリティカルとしてマーク** と **しきい値**（上記参照）。
- **複数フロートパス** と **方式**、**最大パス数**（上記参照）。
- **ラグカレンダー** — 稼働日単位のラグがどのカレンダーを使うか：**先行**（先行タスクのもの）、**後続**（後続タスクのもの）、常に **24時間**、または **プロジェクトカレンダー**。

## 続けて読む

- 複数のクリティカルパス、準クリティカルな作業、Hammock、ハードピン、外部リンクをすべて1つのスケジュールで確認する：[Nieuwbouw Appartementencomplex De Vaart](examples://showcase-appartementencomplex.ifc)。
- 関係、ラグ／リード、制約（ハードピンを含む）については、ガイド [関係と制約](docs://gids-relaties-constraints) でさらに詳しく説明しています。
- 山崩しはクリティカルパスの構造を変えることがあります — ガイド [リソース、ヒストグラム、山崩し](docs://gids-resources-histogram) を読んでください。
- 進捗と状態日は、すでに固定されたタスクに負のフロートを生じさせることがあります — ガイド [ベースラインと進捗](docs://gids-baselines-voortgang) を読んでください。
