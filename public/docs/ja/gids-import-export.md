# インポート/エクスポート

Open Planner Studio は、既定でプロジェクトを IFC として保存します — その隣に別のプロジェクトファイルはありません。ですが、スケジュールをアプリの外でも扱いたいことがあります。Primavera P6、Microsoft Project、あるいはスプレッドシート用のフラットなテーブルとしてです。このガイドでは、ネイティブの IFC フォーマットが実際に何を意味するのか、各エクスポート形式が何を含み何を含まないのか、そしてアプリのどこでインポート／エクスポートを行うのかを説明します。

## ここで学ぶこと

- 「IFC がネイティブフォーマットである」ことが、開く／保存する動作にとって正確に何を意味するか。
- MS Project（MSPDI）と Primavera P6 XML にエクスポートするとき、何が引き継がれ、何が引き継がれないか。
- CSV エクスポートに何が含まれ、意図的に何が除外されているか。
- インポートとエクスポートの場所：**Backstage → エクスポート** と **Backstage → インポート**。
- 拡張機能が追加のインポート形式をどう追加できるか。

## IFC：ネイティブフォーマット

Open Planner Studio のプロジェクトは *それ自体が* IFC 4x3 ファイル（buildingSMART 標準）です。その隣に別の JSON やプロジェクトファイルはありません。**保存** と **開く**（Backstage、または **Ctrl+S**／**Ctrl+O**）は IFC を直接読み書きします。つまり、アプリで行うことすべて — タスク、WBS、制約付きの関係、リソースと割り当て、カレンダー（プロジェクトカレンダーとリソースカレンダーの両方）、ベースライン、進捗、メモ、アクティビティコード、カスタムフィールド、プロジェクト間の外部リンク — が同じファイルに収まり、次に **開く** ときに完全な形で戻ってきます。アプリの中で新しい種類のプロジェクトデータに出会ったら、それは IFC を通じてラウンドトリップすると考えて構いません。もし何かがラウンドトリップ *しない* 場合は、以下でその旨を明示しています。

IFC はまた、このアプリが OpenAEC ツールキットの他の部分とつながる方法でもあります。同じファイルを BIM ソフトウェアが読み取り、4D 連携（建物モデルと並んだスケジュール）に使うことができます。

## 他の形式へのエクスポート

**Backstage → エクスポート** を開くと、4つの形式があります。

- **CSV（セミコロン区切り）** — 汎用的なテーブルエクスポート。日付と期間を含むすべてのタスク。
- **MS Project XML** — Microsoft Project で開けます。完全な WBS 構造。
- **Primavera P6 XML** — Oracle Primavera P6 用。
- **IFC 4x3** — buildingSMART 標準で、ネイティブフォーマットと同じもの（別ファイルへの「名前を付けて保存」として、あるいは開いている他のドキュメントに触れずにコピーを共有するのに便利です）。

各形式にはそれぞれ限界があります。ターゲット形式が豊富であるほど多くが引き継がれますが、3つの外部形式のいずれも IFC を完全に映し出すものではありません。

### CSV

CSV エクスポートには **タスクテーブルのみ** が含まれます。WBS コード、名前、期間（日数）、開始、終了、先行タスク（`2.1FS+3d` のようなテキストコードとして）、タスク種別、状態、進捗（%）、実績開始／実績終了、クリティカル（はい／いいえ）、トータルフロート、説明です。**リソース、割り当て、カレンダー、ベースラインは意図的に除外されています** — CSV は、スプレッドシートでスケジュールを閲覧・編集したい人のための純粋なタスクテーブルであり、完全な忠実度を持つプロジェクト交換形式ではありません。CSV ファイルを **インポート** して戻すと、ベースラインは空のままになります（読み込む元のデータがそもそもなかったためです）。

### MS Project XML（MSPDI）

MSPDI は CSV よりもかなり豊富です。リソース、（負荷曲線を含む）割り当て、カレンダー、ベースラインは引き継がれます。それでも、MSPDI で表現できないものもあります。エクスポート時、アプリは何かが失われるたびに開発者コンソール（`console.warn`）に警告を出し、影響を受けた項目の正確な件数を示します。

- プロジェクト間の **外部リンク** は失われます（相手側タスクの「ゴースト」参照はアプリ内にのみ残ります）。
- **緩やかな Start On/Finish On 制約**（緩やかな `MSO`／`MFO`）は SNET/FNET に格下げされます — MSPDI のコード2/3は *ハード*（Must）扱いのため、緩やかなバリアントの上限は失われます。ハードな `MSO`／`MFO` は正確にエクスポートされます。
- **副制約** は失われます — MSPDI にはタスクごとに1つの制約フィールドしかありません。
- **Hammock タスク**（派生期間）は、計算済みの日付を持つ普通のタスクとしてエクスポートされます — MSPDI にはネイティブの Hammock／LOE 型がありません。
- **タスクメモ** は、MSPDI に `<Notes>` フィールドがあるにもかかわらず、意図的に **エクスポートされません**。当アプリのメモはチェックボックス付きのチェックリスト形式であり、プレーンテキストにきれいには変換できないためです。
- **クリティカルパスの定義**（準クリティカルモード／しきい値）などのスケジューリングオプションは、MSPDI にはネイティブに表現できないため失われます — それらは IFC 経由でのみ保持されます。

### Primavera P6 XML

MSPDI と同種のトレードオフがあり、いくつか P6 固有の癖もあります。

- **外部リンク** と **Hammock タスク** は、MSPDI と同じように失われる／簡略化され、それぞれ警告が出ます。
- **タスクメモ** もここでは除外されます — P6 XML にはそれに適したフィールドがありません。
- 関係の **パーセンテージラグ**（例えば先行タスクの期間の40%）は、P6 にパーセンテージラグの概念がないため、固定の日数に「焼き込まれ」ます。
- **暦日ラグ**（稼働日ではなく経過日数によるラグ）は、単純な時間ベースのラグとしてエクスポートされます — P6 には関係ごとの独立したラグ単位がありません。
- **後期ピーク（LATE_PEAK）** の負荷曲線には P6 に相当するものがなく、最も近い近似値（「Early Peak」）としてエクスポートされます。
- スケジューリングオプションは（MSPDI と同様に）エクスポートされません。

これらの警告は雑さの表れではなく、意図的で明示的な選択です。項目が失われるたびに見える警告を出す方が、静かなデータ損失よりも良いのです。例えば、ショーケース [Nieuwbouw 6 Rijwoningen De Akkers](examples://showcase-rijwoningen-de-akkers.ifc)（タスクメモとパーセンテージラグを持つ関係があります）を開いて P6 または MS Project XML にエクスポートしてみてください。開発者コンソールに、どの項目が失われた、または簡略化されたか、その件数とともに正確に表示されます。

## インポートする

**ファイル → 開く**（または **Backstage → 開く**）は `.ifc`、`.csv`、`.xml` ファイルを受け付けます。`.xml` ファイルの場合、アプリは内容に基づいて、それが Primavera P6 ファイルか MS Project ファイルかを自動的に判別します。上記のとおり、CSV または P6 のインポートは **ベースラインを持たない** プロジェクトを生成します（元データにベースラインがなかったためです）。一方、IFC と MSPDI はベースラインを引き継ぎます。

## 拡張機能インポーター

上記の固定形式に加えて、インストールされた拡張機能は独自のインポーターを追加できます — 例えば既定ではサポートされていない形式のためです。それらは **Backstage → インポート** の下に、それぞれ名前、説明、対応するファイル拡張子とともに表示されます。インポート用の拡張機能が1つもインストールされていない場合、そのセクションは空です。利用可能な拡張機能については **Backstage → 拡張機能** を確認してください。

## さらに詳しく

- ベースラインは IFC と MS Project XML でのみ引き継がれ、CSV や P6 では引き継がれません — ベースラインの記録方法については、ガイド [ベースラインと進捗](docs://gids-baselines-voortgang) を読んでください。
- リソース、割り当て、負荷曲線については、エクスポート前にそれらがどう構築されるかについて、ガイド [リソース、ヒストグラム、山崩し](docs://gids-resources-histogram) を読んでください。
