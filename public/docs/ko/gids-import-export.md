# 가져오기/내보내기

Open Planner Studio는 기본적으로 프로젝트를 IFC로 저장합니다 — 별도의 프로젝트 파일이 함께 있지 않습니다. 하지만
때로는 일정이 Primavera P6, Microsoft Project, 또는 스프레드시트용 평면 표처럼 앱 외부에서도
존재해야 합니다. 이 가이드는 네이티브 IFC 형식이 정확히 무엇을 의미하는지, 각 내보내기 형식이
무엇을 가져가고 가져가지 않는지, 그리고 앱에서 가져오기/내보내기가 어디에 있는지 설명합니다.

## 여기서 배울 내용

- "IFC가 네이티브 형식"이라는 것이 열기와 저장에 정확히 무엇을 의미하는지.
- MS Project(MSPDI)와 Primavera P6 XML로 내보낼 때 무엇이 함께 오고 무엇이 오지 않는지.
- CSV 내보내기에 무엇이 포함되는지 — 그리고 의도적으로 제외된 것.
- 가져오기/내보내기 위치: **백스테이지 → 내보내기**와 **백스테이지 → 가져오기**.
- 확장 프로그램이 추가 가져오기 형식을 어떻게 더할 수 있는지.

## IFC: 네이티브 형식

Open Planner Studio 프로젝트는 *곧* IFC 4x3 파일입니다(buildingSMART 표준). 별도의 JSON이나
프로젝트 파일이 함께 있지 않습니다: **저장**과 **열기**(백스테이지, 또는 **Ctrl+S**/**Ctrl+O**)는
IFC를 직접 읽고 씁니다. 이는 앱에서 하는 모든 것 — 작업, WBS, 제약 조건이 있는 관계, 자원과
할당, 달력(프로젝트 달력과 자원 달력 모두), 기준선, 진행률, 메모, 작업 코드와 사용자 정의
필드, 프로젝트 간 외부 링크 — 이 같은 파일에 저장되며 다음에 **열기**할 때 온전히 돌아온다는
뜻입니다. 앱에서 새로운 종류의 프로젝트 데이터를 만나면, 그것이 IFC를 통해 왕복한다고 가정할
수 있습니다; 만약 왕복하지 *않는* 것이 있다면 아래에 명시적으로 언급됩니다.

IFC는 이 앱이 나머지 OpenAEC 툴킷에 연결되는 방식이기도 합니다: 같은 파일을 BIM 소프트웨어가
읽어 4D 연동(빌딩 모델과 함께하는 일정)에 사용할 수 있습니다.

## 다른 형식으로 내보내기

**백스테이지 → 내보내기**를 열면 네 가지 형식이 있습니다:

- **CSV (세미콜론 구분)** — 범용 테이블 내보내기. 날짜와 기간이 포함된 모든 작업.
- **MS Project XML** — Microsoft Project에서 열립니다. 전체 WBS 구조.
- **Primavera P6 XML** — Oracle Primavera P6용.
- **IFC 4x3** — buildingSMART 표준, 네이티브 형식과 동일합니다(별도 파일로의 "다른 이름으로
  저장"이나, 나머지 열린 문서를 건드리지 않고 사본을 공유하는 데 편리합니다).

각 형식에는 고유한 제한이 있습니다: 대상 형식이 풍부할수록 더 많은 것이 함께 가지만, 세 가지
외부 형식 중 어느 것도 IFC의 완전한 거울상은 아닙니다.

### CSV

CSV 내보내기에는 **작업 표만** 포함됩니다: WBS 코드, 이름, 기간(일), 시작, 종료, 선행 작업(텍스트
코드로, 예: `2.1FS+3d`), 작업 유형, 상태, 완료(%), 실제 시작/종료, 크리티컬(예/아니오), 총 여유,
설명. **자원, 할당, 달력, 기준선은 의도적으로 제외됩니다** — CSV는 스프레드시트에서 일정을
보거나 편집하려는 사람을 위한 순수한 작업 표이지, 완전한 충실도의 프로젝트 교환이 아닙니다.
CSV 파일을 다시 **가져올** 때 기준선은 비어 있게 됩니다(읽어올 것이 없었기 때문입니다).

### MS Project XML (MSPDI)

MSPDI는 CSV보다 상당히 풍부합니다: 자원, 할당(로딩 곡선 포함), 달력, 기준선이 함께 옵니다.
그럼에도 MSPDI로 모든 것을 표현할 수는 없습니다. 내보낼 때 앱은 무언가 손실될 때마다 개발자
콘솔(`console.warn`)에 영향받은 항목의 정확한 개수와 함께 경고를 표시합니다:

- 프로젝트 간 **외부 링크**는 제외됩니다(다른 작업의 "고스트" 참조는 앱 내부에만 유지됩니다).
- **소프트 시작일에/완료일에(Start On/Finish On) 제약 조건**(소프트 `MSO`/`MFO`)은 SNET/FNET으로
  격하됩니다 — MSPDI 코드 2/3은 *하드*(Must)이므로 소프트 변형의 상한이 손실됩니다. 하드
  `MSO`/`MFO`는 정확히 내보내집니다.
- **보조 제약 조건**은 손실됩니다 — MSPDI는 작업당 하나의 제약 조건 필드만 있습니다.
- **hammock 작업**(파생 기간)은 계산된 날짜를 가진 일반 작업으로 내보내집니다 — MSPDI에는 네이티브
  hammock/LOE 유형이 없습니다.
- **작업 메모**는 MSPDI에 `<Notes>` 필드가 있음에도 의도적으로 내보내지지 **않습니다**: 우리의
  메모는 체크박스가 있는 체크리스트 형식이라 일반 텍스트로 깔끔하게 변환되지 않습니다.
- **크리티컬 패스 정의**(준임계 모드/임계값)와 기타 스케줄링 옵션은 MSPDI에서 네이티브로 표현할
  수 없으므로 손실됩니다 — 이들은 IFC를 통해서만 보존됩니다.

### Primavera P6 XML

MSPDI와 같은 종류의 트레이드오프이며, P6 특유의 몇 가지 특이점이 있습니다:

- **외부 링크**와 **hammock 작업**은 MSPDI와 같은 방식으로 제외/단순화되며, 각각 경고와 함께
  표시됩니다.
- **작업 메모**도 여기서 제외됩니다 — P6 XML에는 이를 위한 적절한 필드가 없습니다.
- 관계의 **백분율 지연**(예: 선행 작업 기간의 40%)은 고정된 일수로 "구워집니다", P6에는 백분율
  지연 개념이 없기 때문입니다.
- **달력일 지연**(근무일이 아닌 경과일 단위 지연)은 일반 시간 단위 지연으로 내보내집니다 — P6에는
  관계별로 별도의 지연 단위가 없습니다.
- **후기 피크(LATE_PEAK)** 로딩 곡선은 P6에 대응하는 항목이 없어 가장 가까운 근사값("Early
  Peak")으로 내보내집니다.
- 스케줄링 옵션은(MSPDI와 마찬가지로) 내보내지지 않습니다.

이러한 경고는 부주의함이 아닙니다 — 의도적이고 명시적인 선택입니다: 제외된 항목마다 눈에 보이는
경고가 조용한 데이터 손실보다 낫습니다. 예를 들어 쇼케이스
[Nieuwbouw 6 Rijwoningen De Akkers](examples://showcase-rijwoningen-de-akkers.ifc)(작업 메모와
백분율 지연이 있는 관계가 있음)를 열고 P6나 MS Project XML로 내보내 보세요: 개발자 콘솔은
정확히 어떤 항목이 제외되거나 단순화되었는지, 그리고 몇 개인지 보여줍니다.

## 가져오기

**파일 → 열기**(또는 **백스테이지 → 열기**)는 `.ifc`, `.csv`, `.xml` 파일을 받습니다. `.xml`
파일의 경우, 앱은 내용을 기반으로 Primavera P6 파일인지 MS Project 파일인지 스스로 감지합니다.
위에서 설명했듯이: CSV나 P6 가져오기는 **기준선 없는** 프로젝트를 만듭니다(소스에 없었기
때문입니다), 반면 IFC와 MSPDI는 기준선을 함께 가져옵니다.

## 확장 프로그램 가져오기 도구

위의 고정된 형식 외에도, 설치된 확장 프로그램은 기본적으로 지원되지 않는 형식 등을 위해 자체
가져오기 도구를 추가할 수 있습니다. 이들은 **백스테이지 → 가져오기** 아래에 각각의 이름, 설명,
일치하는 파일 확장자와 함께 나타납니다; 가져오기 확장 프로그램이 설치되어 있지 않으면 해당
섹션은 비어 있습니다. 무엇이 사용 가능한지 확인하려면 **백스테이지 → 확장 프로그램**을
확인하세요.

## 더 읽어보기

- 기준선은 CSV나 P6가 아니라 IFC와 MS Project XML을 통해서만 함께 옵니다 — 기준선을 기록하는
  방법은 가이드 [기준선 및 진행](docs://gids-baselines-voortgang)을 읽어보세요.
- 자원, 할당, 로딩 곡선 — 내보내기 전에 이들이 어떻게 구성되는지는 가이드
  [자원, 히스토그램 및 평준화](docs://gids-resources-histogram)를 읽어보세요.
