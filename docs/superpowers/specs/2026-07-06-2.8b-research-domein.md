## 1. Oracle Primavera P6

### 1.1 Interne opslag = tijd, geen dagen
P6 rekent **native in uren** en converteert alles daarheen: "Primavera P6 functions based on hours
as the standard unit of time … P6 internally converts these to hours for its calculations."
Concreet slaat de database vervolgens **minuten** op: "When I enter a duration of 3d … P6 will
multiply it by 8 Hours per Day to get 24 Hours. It then stores this in the database in minutes as
24 Hours times 60 Minutes is 1440 Minutes." WEB `https://tensix.com/how-long-is-a-primavera-p6-day-week-month-or-year/`,
WEB `https://www.p6consulting.ca/blog/primavera/calendars/primavera-p6-calendars-activity-durations-and-hours-per-time-period-explained/`.
(Sommige bronnen zeggen "seconden vóór opslag"; de precieze diepste sub-eenheid varieert per bron —
de kern **"tijd, niet dagen; minuten als opslageenheid"** is consistent. De seconden-claim zelf is UNVERIFIED.)

### 1.2 Hours per Day/Week/Month/Year = conversiefactoren ("Time Periods")
P6 heeft een aparte **Time Periods**-instelling ("Specify the number of work hours for each time
period") die los staat van de werktijden in de werkweek. Systeemdefaults:
**Day = 8h, Week = 40h, Month = 172h, Year = 2000h.** WEB `tensix.com/how-long-is-a-primavera-p6-day-week-month-or-year/`.
Deze factoren bepalen hoe P6 een ingevoerde duur (in d/w/mo) naar uren omrekent en hoe het duren in
rapporten weer terugtoont. **Kritieke regel**: Hours/Day én Hours/Week moeten **exact** overeenkomen
met de werkelijke werkweek-uren; Hours/Month en Hours/Year zijn "estimates that may vary". Mismatch
tussen werkweek en Time Periods = P6 gebruikt de Time Periods, wat de beruchte "mijn duur veranderde
vanzelf"-fout geeft. WEB `p6consulting.ca/...`, WEB `https://www.planacademy.com/primavera-calendars-change-durations/`.
→ Dit is exact wat de OPS-writer al benadert met `HoursPerDay/HoursPerWeek/HoursPerMonth` (CODE `p6xmlWriter.ts:183-185`),
alleen ontkoppelt OPS de factoren nog niet van de werkweek-banden.

### 1.3 Work periods per weekdag (tijdvakken per dag)
De P6-**werkweek** definieert per dag-van-de-week de werktijden; een dag kan **meerdere
werktijd-blokken** hebben (bv. ochtend- en middagblok rond een lunchpauze). WEB
`https://tensix.com/calendar-time-periods-in-p6-professional/`, WEB `https://tensix.com/calendars-in-oracle-primavera-p6-explained/`
(algemene P6-kalenderdocumentatie; de exacte multi-band-UI is standaard maar niet uit één enkele
geciteerde zin — behandel "meerdere banden per dag" als WEB-algemeen, niet als hard citaat).
De P6-XML draagt dit als `<StandardWorkWeek>` → per dag een `<StandardWorkHour>` met `<DayOfWeek>` en
`<WorkTime>`-blokken (`<Start>`/`<Finish>` in `HH:MM:SS`). CODE `p6xmlWriter.ts:86-100`. **OPS leest
nu alleen het láátste `<WorkTime>`-blok als één start/finish-scalar** — meerdere banden per dag gaan
verloren. CODE `p6xmlReader.ts:92-98`.

### 1.4 Shift calendar vs. resource shifts (twee verschillende dingen)
- **Shift calendar** (tabblad Units & Prices): een **kostenfunctie**. Het bepaalt de Price/Unit
  afhankelijk van wélke ploeg (tijdvenster) de activiteit draait — P6 rekent resourcekosten op basis
  van de shift waarin het werk valt. WEB `https://planningplanet.com/blog/how-does-resource-shift-calendar-primavera-p6-work`,
  WEB `https://kdblog.net/2016/07/19/how-does-resource-shift-calendar-in-primavera-p6-work/`.
- **Resource shifts / resource-kalenders**: definiëren **beschikbaarheid** (histogram-limieten,
  afwijkend rooster van een onderaannemer/materieel), niet de kostprijs. WEB (zelfde bronnen),
  WEB `https://eastwoodharris.com/DL/TP/191101_Scheduling_with_Shifts_in_P6.pdf`.

Belangrijke nuance voor 2.8b: in P6 is een "shift" primair een **kostprijs-/beschikbaarheids-venster**,
niet het mechanisme om dag- vs. nachtwerk te *plannen*. Dag/nacht-plánning doe je in P6 met
**verschillende kalenders** (elk met eigen werktijd-banden) per activiteit/resource. UNVERIFIED-nuance:
of P6 een expliciete "night calendar"-tag kent (nee — het is gewoon een gewone kalender met
nacht-banden); dit volgt uit de kalenderdocumentatie, niet uit één geciteerde bron.

### 1.5 Data date op uur-niveau
De **data date** is het punt tot waar actuals zijn gerapporteerd en vanaf waar toekomstig werk wordt
ingepland. Eén bron stelt: "always starts at the beginning of the day." WEB
`https://www.taradigm.com/understanding-and-preventing-schedule-drift-in-primavera-p6/`.
P6 slaat de data date als **datetime** op (dus mét tijd), maar het exacte default-tijdstip
(middernacht vs. start werkdag) is per bron onduidelijk → **UNVERIFIED**. OPS kapt de data date
sowieso af tot alleen de datum. CODE `p6xmlReader.ts:490-491` (`statusDate = parseP6Date(...)` = `substring(0,10)`).

### 1.6 Lag in uren + lag-kalender-opties
P6-lag wordt intern **in uren** gerekend en kan positief/negatief zijn. Er zijn **vier
lag-kalender-opties**: predecessor-kalender, successor-kalender, 24-uurs-kalender, of
project-default-kalender. WEB `https://www.taradigm.com/...`, WEB `https://tensix.com/scheduling-a-24-hour-elapsed-time-lag-in-primavera-p6/`.
Een **24-uurs elapsed lag** (telt gewone klok-uren, niet werktijd) is de standaardtruc voor bv.
beton-uithardtijd. WEB `tensix.com/scheduling-a-24-hour-elapsed-time-lag-...`.
→ Dit bevestigt de bron-onzekerheids-notitie in OPS (predecessor als beste default). CODE `lagCalendar.ts:1-10`.
OPS rondt lag nu naar hele dagen af (verlies). CODE `p6xmlReader.ts:417`.

## 2. Microsoft Project (MSPDI)

### 2.1 Interne granulariteit = minuten (lag: tienden van minuten)
MS Project converteert duren intern naar uren volgens de Options-defaults en rekent op
**minuut-niveau**: "each hour is comprised of 60 one-minute segments" (minuut = basiseenheid van de
scheduling-engine). WEB `https://mpug.com/how-microsoft-project-calculates-task-duration`.
In het **MSPDI-uitwisselingsformaat** wordt **lag** opgeslagen in **tienden van een minuut**,
ongeacht weergave: "the amount of lag is in tenths of a minute … One work day equals 8 hours, or 480
minutes, or 4,800 tenths of minutes." WEB `https://mavgantt.com/integrations/microsoft-project`.
→ Bevestigt de OPS-codeconventie exact. CODE `mspdiReader.ts:67-72`, CODE `mspdiWriter.ts:51-54`.
UNVERIFIED: of MSP *intern in de applicatie* ook in tienden-van-minuten opslaat (de bron bevestigt dit
alleen voor de MSPDI-lag-encoding; de engine-basis lijkt de hele minuut).

### 2.2 WeekDays / WorkingTimes-structuur in MSPDI-XML
Kalender = `<Calendar>` met `<WeekDays>` → per dag een `<WeekDay>` met `<DayType>` (MSP-nummering
1=zondag..7=zaterdag), `<DayWorking>` (0/1) en, indien werkdag, `<WorkingTimes>` → **één of meer**
`<WorkingTime>`-blokken met `<FromTime>`/`<ToTime>`. WEB `https://learn.microsoft.com/en-us/office-project/xml-data-interchange/task-elements-and-xml-structure`
(MSPDI-schema-documentatie). CODE-bevestiging van de structuur die OPS schrijft/leest:
`mspdiWriter.ts:90-106`, `mspdiReader.ts:421-460`. **OPS schrijft nu maar één band (07-16) en leest
alleen het eerste blok** — meerdere `<WorkingTime>`-banden (bv. lunchpauze) gaan verloren. CODE `mspdiReader.ts:446-460`.

### 2.3 DurationFormat / LagFormat-codes
Beide zijn subsets van dezelfde MSPDI-enum: **3=m, 4=em, 5=h, 6=eh, 7=d, 8=ed, 9=w, 10=ew, 11=mo,
12=emo, 19=%, 20=e%** (prefix `e` = **elapsed** / 24-7; suffix `?` = geschat, codes 35+). WEB
`https://learn.microsoft.com/en-us/office-project/xml-data-interchange/durationformat-element`,
WEB `mavgantt.com/integrations/microsoft-project`.
→ OPS gebruikt precies deze codes: 7=dagen, 8=elapsed dagen, 19=%, 20=elapsed% bij schrijven; reader
herkent elapsed-set {4,6,8,10,12} en %-set {19,20}. CODE `mspdiWriter.ts:56-68`, CODE `mspdiReader.ts:307,319-327`.

### 2.4 Uren→dagen-presentatie (defaults 8h=1d, 40h=1w)
MS Project toont duren in een gekozen eenheid en rekent om via de Options-defaults **8h/dag,
40h/week**. WEB `https://support.microsoft.com/en-us/office/change-the-default-task-duration-unit-...`,
WEB `mpug.com/how-microsoft-project-calculates-task-duration`. Duur-element in ISO 8601:
`PT8H0M0S` = 8u. WEB `learn.microsoft.com/.../task-elements-and-xml-structure`. Deze defaults zitten in
MSPDI als `<MinutesPerDay>`/`<MinutesPerWeek>`; OPS schrijft ze uit `hoursPerDay`. CODE `mspdiWriter.ts:169-171`.

### 2.5 CalendarUID op taken
MSPDI ondersteunt **taak-kalenders native** via `<CalendarUID>` op `<Task>` (en op `<Resource>`).
CODE `mspdiWriter.ts:241-246`, CODE `mspdiReader.ts:198-201`. Dit is de MSP-manier om één taak op een
afwijkend (bv. nacht-)rooster te zetten — direct bruikbaar voor OPS-ploegen.

## 4. IFC 4.3 (buildingSMART)

### 4.1 IfcWorkTime — Start/Finish + recurrence
`IfcWorkTime` (subtype van `IfcSchedulingTime`) heeft `RecurrencePattern`, `Start` en `Finish`. **Start
= de datum om 0:00, Finish = de datum om 24:00**; een recurrence-pattern, indien aanwezig, "is valid
within the time period defined by IfcWorkTime.Start and IfcWorkTime.Finish." Zonder recurrence beslaat
het blok de hele periode van middernacht tot middernacht (bruikbaar voor feestdag-ranges). WEB
`https://github.com/buildingSMART/IFC4.3.x-development/blob/master/docs/schemas/resource/IfcDateTimeResource/Entities/IfcWorkTime.md`.
→ OPS gebruikt precies dit: werkweek als `IFCWORKTIME(…recurrence…)`, feestdagen als
`IFCWORKTIME(name,…,'start','end')` zónder recurrence. CODE `ifcWriter.ts:477,483`.

### 4.2 IfcRecurrencePattern — meerdere tijdvakken per dag zijn native
`IfcRecurrencePattern` heeft o.a. `RecurrenceType`, `WeekdayComponent` (set weekdagen), `Interval`,
`Position`, en cruciaal **`TimePeriods`**: een optionele **lijst** van `IfcTimePeriod`, "enabling time
periods such as **between 7:00 and 12:00 and between 13:00 and 17:00** for each of the applicable days,
weeks or months." WEB `https://ifc43-docs.standards.buildingsmart.org/IFC/RELEASE/IFC4x3/HTML/lexical/IfcRecurrencePattern.htm`.
→ IFC ondersteunt dus **native meerdere werktijd-banden per dag** (echte lunchpauze). **OPS schrijft nu
maar één `IFCTIMEPERIOD` per kalender.** CODE `ifcWriter.ts:475-476`.

### 4.3 IfcTimePeriod / IfcTime — tijd-van-de-dag
`IfcTimePeriod` = start- en eindtijd, beide `IfcTime` (tijd-van-de-dag, mét tijdzone en DST-offset).
`IfcTime` is een IFC4-type dat het oude `IfcLocalTime`/`IfcCalendarDate`/`IfcDateAndTime` vervangt. WEB
`https://ifc43-docs.standards.buildingsmart.org/IFC/RELEASE/IFC4x3/HTML/lexical/IfcTimePeriod.htm`.

### 4.4 IfcTaskTime — granulariteit (uren zijn native mogelijk)
`IfcTaskTime.ScheduleDuration` is een **`IfcDuration`** = ISO 8601 lexicaal `PnYnMnDTnHnMnS`, dus **kan
uren dragen** (bv. `PT5H`). De duur is elapsed óf werktijd, bepaald door `DurationType`. WEB
`https://ifc43-docs.standards.buildingsmart.org/IFC/RELEASE/IFC4x3/HTML/lexical/IfcTaskTime.htm`,
WEB `https://standards.buildingsmart.org/IFC/RELEASE/IFC4/ADD1/HTML/schema/ifcdatetimeresource/lexical/ifcduration.htm`.
Start/Finish etc. zijn `IfcDateTime` (datum **+ tijd**). → **OPS schrijft duur bewust als `P0Y0M{days}D`
(alleen dagen) en datetimes met vast `T07:00:00`** — een verlieslatende dag-subset van wat IFC aankan.
CODE `ifcWriter.ts` (`ifcDuration`, `ifcDateTime`).

### 4.5 PredefinedType FIRSTSHIFT/SECONDSHIFT/THIRDSHIFT — semantiek
`IfcWorkCalendar.PredefinedType` is `IfcWorkCalendarTypeEnum` = **FIRSTSHIFT, SECONDSHIFT, THIRDSHIFT,
USERDEFINED, NOTDEFINED**. WEB `https://ifc43-docs.standards.buildingsmart.org/IFC/RELEASE/IFC4x3/HTML/lexical/IfcWorkCalendarTypeEnum.htm`,
WEB `https://ifc43-docs.standards.buildingsmart.org/IFC/RELEASE/IFC4x3/HTML/lexical/IfcWorkCalendar.htm`.
Constraint `CorrectPredefinedType`: bij `USERDEFINED` moet `ObjectType` gezet zijn. **De buildingSMART-
documentatie definieert NIET wat FIRSTSHIFT/SECONDSHIFT/THIRDSHIFT semantisch betekenen** — de
IfcWorkCalendar-pagina bevat geen tekst over ploegpatronen. WEB (IfcWorkCalendar-pagina, expliciet
gecontroleerd: "documentation does not explicitly define what FIRSTSHIFT/SECONDSHIFT/THIRDSHIFT
represent"). De gangbare interpretatie (1e=dagdienst, 2e=avond/swing, 3e=nacht) is **industrie-
conventie, niet spec** → **UNVERIFIED**. Praktisch: `PredefinedType` is een **label voor wélke ploeg de
kalender vertegenwoordigt**; de werkelijke tijdvakken zitten in `WorkingTimes`. OPS zet dit nu
hardcoded op `.FIRSTSHIFT.` voor élke kalender. CODE `ifcWriter.ts:489`.

## 5. Conventies — afronding & weergave

**Gedeelde conventie (P6 én MSP): grondslag ≠ weergave.** Beide bewaren duur intern in tijd
(P6=minuten, MSP=minuten/tienden) en **tonen** die in een door de gebruiker gekozen eenheid via
conversiefactoren (P6 Time Periods 8/40/172/2000; MSP Options 8h/40w). WEB §1.2, §2.4.

**De "mixed calendar"-val.** Omdat P6 een duur toont als `interne_uren ÷ Hours/Day-van-de-kalender`,
laat P6 twee activiteiten met verschillende kalenders (bv. 8u vs. 10u/dag) allebei als "1d" zien
terwijl ze verschillend veel klok-uren beslaan — en een duur die je met "d" invoert kan van waarde
lijken te veranderen als de kalender-Hours/Day niet klopt. WEB `planacademy.com/primavera-calendars-change-durations/`,
WEB `p6consulting.ca/...`. Voor een planner die dag- én uur-projecten mengt betekent dit: **de getoonde
"1d" is dubbelzinnig zonder de bijbehorende Hours/Day**; de kalender-context moet zichtbaar zijn.

**Elapsed vs. work duration.** Beide tools kennen "elapsed" duren/lags die 24-uurs klok-tijd tellen
i.p.v. werktijd (P6 24h-kalender/elapsed lag §1.6; MSP `e`-prefix DurationFormat §2.3). OPS heeft dit
al als `durationType: 'WORKTIME' | 'ELAPSEDTIME'` (CODE `task.ts:34,45`) en `lagUnit: 'ELAPSEDTIME'`
in de adapters (CODE `mspdiWriter.ts:58-67`).

**Afronding — huidig OPS-verlies.** Beide OPS-readers doen `Math.round(...)` naar hele dagen bij import
(CODE §0). Een 4-uurstaak wordt daardoor 0 of 1 dag; een 12-uurs lag verdwijnt. De industrie rondt
**alleen voor weergave**, nooit in de opslag. Dit is precies de conventie die 2.8b moet overnemen.

## 6. Aanbevelingen voor OPS 2.8b

### 6.1 Grondslag-eenheid → integer **minuten**
Beide leidende tools bewaren tijd, geen dagen; de kleinste gedeelde noemer is de **minuut** (P6=minuten,
MSP=minuten met tienden alleen in de MSPDI-lag-encoding). Aanbeveling: introduceer een interne
**minuten-grondslag** voor duur en lag (integer minuten; vermijdt float-drift die uren-met-decimalen
geven). `scheduleDuration` blijft naar buiten toonbaar als werkdagen, maar de **bron van waarheid wordt
minuten**.
- **Regressie-eis (hard):** de 290 dag-cases mogen niet breken. Ontwerp de minuten-engine zó dat een
  integer-dag-invoer met de default-kalender (single band, `hoursPerDay=8`) **bit-identieke** datums
  oplevert als nu. Concreet: `minuten = dagen × hoursPerDay × 60`, en de dag-tellus is een exacte
  reductie van de minuten-tellus wanneer elke werkdag precies `hoursPerDay×60` minuten telt. Bouw dit
  als conversielaag bovenop `CalendarEngine`, niet als herschrijving, en houd de dag-cases als
  gouden-referentie.
- Alternatief (afgewezen): uren-met-decimalen — leesbaarder maar introduceert afrondingsklassen die
  P6/MSP juist met integer-minuten vermijden.

### 6.2 Kalendermodel → per-weekdag werktijd-**banden** + losgekoppelde conversiefactoren
Vervang de ene `workStartHour`/`workEndHour`-scalar door **een lijst van {start,end}-banden per ISO-
weekdag**. Dit is wat P6 (`StandardWorkHour`/`WorkTime` §1.3), MSP (`WeekDay`/meerdere `WorkingTime`
§2.2) en IFC (`IfcRecurrencePattern.TimePeriods`-lijst §4.2) **allemaal native dragen**, en het maakt
een echte lunchpauze modelleerbaar (07:00-12:00 + 12:30-16:30) i.p.v. de huidige impliciete pauze
(07-16 klok, `hoursPerDay=8`).
- Maak `hoursPerDay` **afgeleid** (som van de bandlengtes), maar behoud een **expliciete
  conversiefactor-set** (P6 Time Periods: uren/dag/week/maand/jaar) voor duur-weergave, ontkoppeld van
  de werkelijke banden — precies zoals P6 die twee scheidt (§1.2). Dag/week moeten kloppen; maand/jaar
  zijn schattingen.
- **Ploegen (dag/nacht) → één kalender per ploeg.** Kies het P6/MSP-model (aparte kalenders per ploeg,
  toegekend per taak/resource via de bestaande `task.calendarId`/`CalendarUID`/`CalendarObjectId`)
  boven het Asta-work-pattern-model. Reden: minimale engine-wijziging, schone adapter-mapping, en het
  maakt de nu-hardcoded IFC-`.FIRSTSHIFT.` eindelijk betekenisvol (schrijf de echte ploeg-enum per
  kalender). Een nachtploeg = een kalender met banden ~22:00-06:00 (let op de dag-overschrijding: een
  band die middernacht kruist moet de engine als twee sub-banden of als "eindigt op volgende
  kalenderdag" behandelen — expliciet ontwerpen).
- **Echt 24-uurs** = een kalender met één band 00:00-24:00 op alle zeven dagen ⇒ `hoursPerDay=24`,
  geen weekend-gaten. Sluit aan op P6's 24h-kalender en Asta's 24u-shift.
- **Lag-kalender**: houd `predecessor` als default (bevestigd: 4 P6-opties, predecessor is de gangbare
  fabrieksdefault — §1.6), stel de keuze pas in 2.9 open zoals gepland. Maar verhuis **lag zelf mee naar
  minuten** (nu naar dagen afgerond, lossy).

### 6.3 Weergaveconventies → minuten intern, dagen/uren getoond, kalender-context zichtbaar
- Toon duur default in **werkdagen** (via de effectieve-kalender-`hoursPerDay`), met een **uur-overlay**
  in de Gantt zodra de uur-tier actief is (`pickTiers` levert die al — CODE `timelineTiers.ts:70-71`).
  Geef een **duur-eenheid-toggle (d/h)** zoals beide tools.
- **Mixed-project-legibiliteit:** omdat "1d" dubbelzinnig is bij verschillende kalender-`hoursPerDay`
  (§5, de P6-val), toon de kalender-`hoursPerDay` in een kolom of tooltip zodra een project kalenders
  met verschillende daglengtes mengt. Waarschuw (net als de bestaande adapter-`console.warn`s) wanneer
  een dag-project en een uur-project in één plan samenkomen.
- **Stop met afronden bij import.** Vervang de `Math.round(...)`-naar-dagen in beide readers door
  minuut-precisie; rond uitsluitend voor weergave.

### 6.4 Adapter-mapping (concreet per adapter)
- **P6** (`p6xmlReader.ts`/`p6xmlWriter.ts`): `PlannedDuration`(uren) ↔ minuten (÷60/×60), geen
  dag-afronding meer; schrijf echte `HoursPerDay/Week/Month/Year` Time Periods; per dag mogelijk
  **meerdere** `<WorkTime>`-banden (lees ze allemaal, niet alleen het laatste blok — fix `p6xmlReader.ts:92-98`);
  lag in uren met minuut-precisie; ploeg via aparte `<Calendar>`-elementen.
- **MSPDI** (`mspdiReader.ts`/`mspdiWriter.ts`): `Duration PT{H}H{M}M0S` ↔ minuten; `MinutesPerDay/
  MinutesPerWeek` blijven (al correct uit `hoursPerDay*60`); schrijf **alle** `<WorkingTime>`-banden
  (nu maar één); **fix de hardcoded `/8` in `parseMSPDuration`** → gebruik `hoursPerDay` (CODE
  `mspdiReader.ts:51`); lag blijft in tienden-van-minuten (encoding klopt al — voed er echte minuten
  in); ploeg via `CalendarUID` (al ondersteund).
- **IFC** (`ifcWriter.ts`): `ifcDuration` moet `PnDTnHnMnS` emitteren i.p.v. `P0Y0M{days}D` (uren native,
  §4.4); `IfcRecurrencePattern.TimePeriods` = **lijst** van de per-dag-banden i.p.v. één `IFCTIMEPERIOD`
  (§4.2, fix `ifcWriter.ts:475-476`); `IfcWorkCalendar.PredefinedType` = de **echte ploeg-enum**
  (FIRST/SECOND/THIRDSHIFT of USERDEFINED+ObjectType) i.p.v. hardcoded `.FIRSTSHIFT.` (§4.5, fix
  `ifcWriter.ts:489`); `IfcTaskTime`-datetimes dragen de **werkelijke tijd-van-de-dag** i.p.v. vast
  `T07:00:00` (§4.4). Markeer de FIRST/SECOND/THIRDSHIFT→dag/avond/nacht-mapping in code als conventie
  (UNVERIFIED per spec), niet als door buildingSMART gedefinieerd.

### 6.5 Buiten scope (bevestigd)
De **instelbare lag-kalender** (P6's 4 opties als keuze) blijft fase 2.9 — de default `predecessor`
volstaat voor 2.8b. CODE `lagCalendar.ts:1-10`, TODO r217. Asta heeft geen adapter en blijft puur
referentie.
