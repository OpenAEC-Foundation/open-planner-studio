## 0. De kern: waar dag-granulariteit fysiek wordt afgedwongen

Twee functies in `src/utils/dateUtils.ts` zijn de **fundamentele bottleneck**. Álle datums in de hele applicatie (engine, adapters, UI, tests, serialisatie) passeren ze, en beide gooien de tijd-van-de-dag weg:

- `dateUtils.ts:2-5` — `parseDate`:
  ```ts
  export function parseDate(iso: string): Date {
    const d = new Date(iso);
    return new Date(Date.UTC(d.getFullYear(), d.getMonth(), d.getDate()));
  }
  ```
  Elke geparste datum wordt **naar middernacht UTC afgekapt**. Uren/minuten in de invoer-ISO gaan onherroepelijk verloren.
- `dateUtils.ts:8-10` — `formatDate`: `d.toISOString().split('T')[0]` → schrijft alléén `YYYY-MM-DD`; tijd wordt weggegooid bij serialisatie.
- Ondersteunende helpers zijn allemaal heel-dag: `addCalendarDays` (`:30-34`, `setUTCDate(...+days)`), `diffCalendarDays` (`:24-27`, `/86400000` afgerond), `isoDayOfWeek` (`:18-21`), `addBusinessDays` (`:91-111`, heel-dag-lus).

**Wat breekt bij uren:** zolang `parseDate`/`formatDate` bestaan, kan géén enkele laag sub-dag-precisie vasthouden — een uur-native datum wordt bij de eerste `parseDate` teruggezet naar 00:00. Dit is de spil van de hele migratie.

**Risico: HOOG.** Dit is het architectuurbesluit dat alles bepaalt (zie Ontwerpvraag 1). Elke andere bevinding hieronder is een symptoom of afgeleide hiervan.

## 1. Engine

### 1.1 `src/engine/scheduler/CalendarEngine.ts` — volledig dag-granulair
De engine kent **geen enkel uur-begrip in zijn rekenlogica**. `hoursPerDay` bestaat wel maar is puur een doorgeefluik:

- `:32-34` `hasWorkingDays()` — kijkt alleen naar `workDays.length` (dagnummers).
- `:37-42` `isWorkDay(date)` — `isoDayOfWeek` + holiday-set; geen tijd.
- `:19-29` `buildHolidaySet()` — feestdagen als **dag-strings** (`formatDate`), inclusieve dag-ranges.
- `:54-75` `addWorkDays`, `:80-90` `workDaysBetween`, `:95-116` `nextWorkDay`/`nextWorkDayAfter`, `:123-146` `prevWorkDay`/`prevWorkDayBefore`, `:152-172` `subtractWorkDays`, `:185-197` `addWorkingDaysSigned` — **allemaal `addCalendarDays(current, ±1)`-lussen over hele dagen**. "Eerste werkdag telt als dag 1" is een dag-conventie.
- `:199-201` `get hoursPerDay()` — leest `calendar.hoursPerDay`, maar **niets in deze klasse gebruikt het**. Puur informatief, precies zoals de 2.8a-handoff meldde.
- `:10-11` `MAX_SCAN = 366` / `MAX_DAYS = 200_000` — dag-tellingen als veiligheidsgrens.

**Wat breekt bij uren:** de hele meeteenheid is "1 werkdag = 1 kalenderdag-stap". Voor uren moet er een tweede notie bij: "binnen een werkdag lopen `hoursPerDay` uren tussen `workStartHour` en `workEndHour`". `addWorkDays(start, 8h)` moet dan óf naar dezelfde dag óf naar de volgende ochtend rollen afhankelijk van de shift-grenzen. Dit is het hart van de 2.8b-implementatie.

**Risico: HOOG** — nieuwe of dual-mode engine nodig.

### 1.2 `src/engine/scheduler/CPMSolver.ts` — integer werkdagen overal
- `:402-403` — kern-duurstap:
  ```ts
  const duration = task.isMilestone ? 0 : task.time.scheduleDuration;
  const earlyFinish = cal.addWorkDays(earlyStart, duration);
  ```
  `scheduleDuration` is integer werkdagen; `addWorkDays` telt dagen.
- `:49-55` `resolveEffectiveLagDays` — `Math.round((predDur * lagPercent)/100)`: procent-lag **op hele dagen afgerond**.
- `:573-576` `resolveLag` — geeft `{days, unit}`; alles in dagen.
- `:616-667` `getForwardConstraint` / `:718-790` `getBackwardConstraint` — relatie-logica (FS/SS/FF/SF): elapsed-lag via `addCalendarDays` (kalenderdagen), worktime-lag via `addWorkingDaysSigned` (werkdagen), successor-mijlpaal-snaps in hele werkdagen. De hele "+1 werkdag ná finish"-conventie is dag-gebonden.
- `:386` in-progress remaining: `Math.round(t.scheduleDuration * (1 - t.completion))` — integer restdagen.
- `:392` `cal.addWorkDays(remStart, remaining)` — restwerk in dagen.
- `:457-462` `signedWorkDays` en `:801-820` relatie-vrije-speling (`workDaysBetween(...) - 1`), `:908` `projectDuration = workDaysBetween(projStart, projectEnd)` — **alle float/duur-uitvoer in integer werkdagen**.
- `:890-893` resultaat-datums via `formatDate(...)` → `YYYY-MM-DD`.

**Wat breekt bij uren:** de solver zelf is grotendeels "engine-agnostisch" (hij delegeert alle datumrekenkunde aan `CalendarEngine`), dus als de engine uren leert, erft de solver dat grotendeels gratis. MAAR: (a) `scheduleDuration` is nu integer — een uur-duur (bv. 4u = 0,5 dag) kan er niet in; (b) alle `- 1`/`+1`-dagconventies in de relatie-snaps veronderstellen "dag als atoom"; (c) float wordt als integer werkdagen naar buiten gerapporteerd en getoond.

**Risico: HOOG** (duur-eenheid + relatie-snap-conventies).

## 2. Datamodel — wat een overstap naar "minuten als grondslag" raakt

### 2.1 `src/types/task.ts` — `TaskTime`
```ts
scheduleDuration: number; // in work days           (:45)
scheduleStart: string;    // ISO 8601               (:47)  ← in de praktijk date-only
scheduleFinish: string;   // ISO 8601               (:48)
earlyStart/earlyFinish/lateStart/lateFinish: string (:51-54)
freeFloat/totalFloat: number; // work days          (:55-56)
actualDuration?/remainingTime?: number              (:62-63)  ← dagen
completion: number; // 0.0 - 1.0                    (:64)
```
- `createDefaultTaskTime` (`:110-139`) leidt `scheduleFinish` af met `addBusinessDays(startDate, durationDays)` — heel-dag.
- `durationType: 'WORKTIME' | 'ELAPSEDTIME'` (`:34`) bestaat al maar heeft niets met uren te maken (werk- vs kalenderdagen).

**Overstap-impact:** dit is het centrale datamodel. Een minuten/uur-grondslag raakt: `scheduleDuration` (getal-eenheid), alle 6 datumvelden (string-formaat), 2 float-velden, `actualDuration`/`remainingTime`. Elk consumeert nu "werkdagen" of "date-only ISO".
**Risico: HOOG.**

### 2.2 `src/types/calendar.ts` — `WorkCalendar`
```ts
workDays: number[];      // 1..7                     (:9)
workStartHour: number;   // e.g. 7                   (:10)
workEndHour: number;     // e.g. 16                  (:11)
hoursPerDay: number;     // net working hours         (:12)
holidays: Holiday[];                                  (:13)
```
De uur-velden **bestaan al** (7-16, 8u) maar zijn informatief — de engine leest ze niet (§1.1). `createDefaultCalendar` (`:41-55`) zet ma-vr 07:00-16:00, 8u.

**Overstap-impact:** de velden zijn er, maar er is **geen ondersteuning voor pauzes, meerdere shift-blokken per dag, of dag/nacht-ploegen** (een nachtploeg 22:00-06:00 kruist middernacht — dat kan het huidige `workStartHour<workEndHour`-model niet uitdrukken). Ploegen-kalender = nieuw datamodel-werk.
**Risico: MIDDEL–HOOG** (velden bestaan, maar shift-model ontbreekt volledig).

### 2.3 `src/types/sequence.ts` — lag
- `:19` `lagDays: number` (positief=uitloop, negatief=lead) — integer dagen.
- `:23-28` `lagPercent` — "afgerond op hele dagen (Math.round; de engine is dag-granulair)".
- `:12` `lagUnit: 'WORKTIME' | 'ELAPSEDTIME'`.
**Overstap-impact:** lag moet sub-dag kunnen (bv. "start 2 uur na"). `lagDays` en de `Math.round` blokkeren dat.
**Risico: MIDDEL.**

### 2.4 `src/types/project.ts` — datums
- `:8-9` `startDate`/`endDate` ISO-datum; `:25` `statusDate?` met comment **"ISO date (dag-granulair)"**; `:27` `progressMode`.
**Overstap-impact:** statusdatum (P6 data date) is expliciet dag-granulair; uur-voortgang vereist een uur-statusdatum.
**Risico: MIDDEL.**

## 4. Adapters — waar de dag↔uur-conversie nu zit, wat verloren gaat, en de `.FIRSTSHIFT.`-hardcode

### 4.1 IFC (`src/services/ifc/`) — het native opslagformaat
De **native save is IFC** (`src/state/slices/fileSlice.ts:1,151,184,243` `writeIFC(...)`, default `*.ifc`). Backward-compat van "het bestandsformaat" = de IFC-round-trip.

- **Duur schrijven** `ifcWriter.ts:44-45` `ifcDuration(days)` → `'P0Y0M${days}D'` — **integer dagen, nul uur-component**.
- **Datum schrijven** `ifcWriter.ts:37-42` `ifcDateTime`: bij een date-only string **`return '${iso}T07:00:00'` — de tijd 07:00 wordt verzonnen**, niet uit data. Alle IfcTaskTime-datumslots (`:589`) gaan hierdoor.
- **`.FIRSTSHIFT.`-hardcode** `ifcWriter.ts:489`:
  ```ts
  `IFCWORKCALENDAR(...,${exceptStr},.FIRSTSHIFT.)`
  ```
  De `IfcWorkCalendarTypeEnum` staat vast op `.FIRSTSHIFT.` — dag/nacht-shift wordt niet uitgedrukt. (Precies de 2.8a-out-of-scope-notitie.)
- **Duur lezen** `ifcReader.ts:172-185` `parseDurationDays`: `P..D` → dagen; **`PT..H` → `Math.ceil(h/8)` met hardcoded 8u/dag** (`:185`). Sub-dag → naar boven afgerond op hele dagen.
- **Datum lezen**: `ifcReader` parst datetimes via `parseDate` → **tijd-van-de-dag gestript** (dus de zojuist geschreven `T07:00:00` gaat er bij inlezen weer af).

**Verlies:** IFC kan nu geen sub-dag dragen (duur = hele `P..D`, tijd hardcoded + weer gestript, shift = `.FIRSTSHIFT.`).
**Risico: HOOG** (native formaat; round-trip-compat + shift-enum).

### 4.2 MSPDI (`src/services/msproject/`)
- **Schrijven** `mspdiWriter.ts:36-39` `durationToISO8601(days, hoursPerDay)` → `PT${days*hoursPerDay}H0M0S` — **gebruikt `calendar.hoursPerDay` (native uren uit!)**, correct maar nooit sub-dag omdat `days` integer is. `formatMSPDateTime` (`:29-32`) hardcodeert **`T08:00:00`**.
- **Lezen** `mspdiReader.ts:44-54` `parseMSPDuration`: **`PT..H → Math.round(h/8)` met HARDCODED 8** (negeert `calendar.hoursPerDay`, inconsistent met de writer én met de p6-reader) → sub-dag verloren. `tenthsOfMinutesToDays` (`:67-72`, lag) gebruikt `hoursPerDay` maar `Math.round` → hele dagen. Datum: `substring(0,10)` (`:41`) **stript tijd**.

**Verlies:** import rondt uren op hele dagen af; de hardcoded 8 in de reader is bovendien een latente bug bij niet-8u-kalenders.
**Risico: MIDDEL.**

### 4.3 P6 XML (`src/services/p6/`)
- **Schrijven** `p6xmlWriter.ts:74-75` `durationToP6Hours(days, hoursPerDay)` → `days * hoursPerDay` (native uren uit); `:283` `PlannedDuration`, `:297` `RemainingDuration`, `:336` `Lag` — allemaal in uren. `HoursPerDay`/`HoursPerWeek` geschreven (`:183-185`).
- **Lezen** `p6xmlReader.ts:66-68` `p6HoursToDays(hours, hoursPerDay)` → **`Math.round(hours / hoursPerDay)`** — rondt naar **hele dagen**. Toegepast op `PlannedDuration` (`:333`), `RemainingDuration` (`:331`), `Lag` (`:417`). Shift-uren `workStartHour/workEndHour` worden wél gelezen (`:94-97`) maar informatief.

**Verlies:** P6-bestanden zijn **uur-native op schijf**; onze reader rondt bij import naar integer dagen → dit is het **grootste verliespunt** voor uur-data uit P6. De writer kan al uren produceren (dag×hoursPerDay), dus zodra intern duren sub-dag kunnen, is P6-export bijna gratis.
**Risico: MIDDEL–HOOG** (P6 is de primaire uur-native interop).

## 6. State / undo / hydrate — serialisatie van datums

- **Snapshot** `src/state/snapshot.ts:38-53` — `createSnapshot` doet `JSON.parse(JSON.stringify(...))` op tasks/sequences/resources/assignments/calendars/baselines. Datums zijn **strings**, dus serialisatie is triviaal en transporteert **elke** string ongewijzigd — óók een toekomstig `"...T14:30"`. Het probleem zit niet in het opslaan maar in het **her-parsen** (elke `parseDate` stript de tijd, §0).
- **DocumentPayload** `src/state/slices/documentSlice.ts:34-61` — zelfde string-datums per document; `capturePayload`/`hydratePayload` (`:116`,`:167`) verplaatsen ze heen en weer. `syncProjectCalendar` (`:193`) na hydrate.
- **Undo/redo** bewaart `cpmResult` per referentie (`snapshot.ts:23-27`) — bevat een `Map`, niet JSON-kloonbaar; blijft werken zolang runCPM het object als geheel vervangt.
- **localStorage** — geen projectdatums; wél persistente view/settings incl. `enableQuarterHourZoom` (`settingsStore.ts:66`) en `timeScale`. Recent-files-lijst (`fileSlice.ts:60`). Geen migratie nodig voor datums; wél als er een nieuwe uur-setting/`timeScale`-preset bijkomt.

**Wat breekt bij uren:** verrassend weinig aan de *serialisatie*-kant (strings gaan door). Het echte risico is dat élk her-hydrate-pad door `parseDate` loopt.
**Risico: LAAG** (serialisatie zelf) — mits §0 wordt opgelost.

## 8. Ontwerpvragen die hieruit volgen

1. **Minuten-als-grondslag vs. dual-mode engine.** Twee wegen:
   (a) **Uniforme minuten-grondslag** — vervang `parseDate`/`formatDate` door datetime-bewuste varianten en maak `CalendarEngine` intern uur/minuut-native (een dag = `hoursPerDay` werkuren). Elegant, maar raakt álle 290 cases en de hele round-trip in één keer.
   (b) **Dual-mode** — dag-engine blijft de default (byte-identiek, 290 groen), uur-engine erbij die alleen aanslaat als een taak/kalender sub-dag-data heeft. Veiliger voor regressie, maar twee codepaden om synchroon te houden.
   *Aanbeveling om te beslissen:* welke geeft de sterkste garantie dat de 290 niet bewegen tegen de laagste dubbel-onderhoudslast?

2. **Duur-eenheid in het datamodel.** Wordt `scheduleDuration` een `number` in **minuten/uren** (met een aparte `durationUnit`), of blijft het "werkdagen" met een optioneel sub-dag-veld? Dit bepaalt de migratie van elke `scheduleDuration`-consument (§3.1) en de baseline-`duration`. Wat is de canonieke atomaire eenheid — minuten (P6/MSPDI rekenen in tienden-van-minuten resp. uren)?

3. **Datum-representatie.** Blijven datums `YYYY-MM-DD`-strings met een apart tijd-veld, of worden het volledige ISO-datetimes (`YYYY-MM-DDTHH:mm`)? Het laatste is minimaal-invasief voor serialisatie (§6) maar vereist dat `parseDate` de tijd behoudt (§0) en dat élke date-only-lezer een default-tijd (shift-begin?) krijgt.

4. **Shift-/ploegen-model.** Hoe worden dag/nacht-ploegen uitgedrukt? Het huidige `workStartHour<workEndHour` kan geen middernacht-kruisende nachtploeg (22:00-06:00) of pauzes/meerdere blokken per dag aan (§2.2). Nodig: een `shifts: {start,end}[]`-achtige structuur op `WorkCalendar` + IFC-mapping weg van de hardcoded `.FIRSTSHIFT.` (naar `.SECONDSHIFT.`/`.THIRDSHIFT.`/user-defined).

5. **Backward-compat bestandsformaten.**
   - **IFC (native):** hoe blijft een bestaand `.ifc` zonder uur-data identiek inlezen, terwijl nieuwe bestanden echte tijden (i.p.v. hardcoded `T07:00:00`) en sub-dag-`PT..H`-duren dragen? Round-trip-stabiliteit is een harde eis.
   - **P6/MSPDI:** de import-readers ronden nu naar hele dagen (`Math.round`/`Math.ceil`). Zodra intern sub-dag mag, moeten deze de uur-precisie **behouden** i.p.v. afronden — en de mspdi-reader-hardcode `/8` (`mspdiReader.ts:50`) vervangen door `calendar.hoursPerDay`.

6. **Backward-compat localStorage/settings.** Komt er een uur-`timeScale`-preset en/of vervangt uur-zoom de bestaande `enableQuarterHourZoom`-gating? Hoe migreren bestaande persistente settings zonder de instelling te resetten?

7. **Regressie-borging.** Kunnen de 290 dag-cases **letterlijk ongewijzigd** blijven (missende uur-velden ⇒ oud gedrag), of moeten ze her-geannoteerd worden? De veiligste eis: geen enkele regel in `cases-*.json` verandert, en de nieuwe uur-cases komen in een aparte batterij met een datetime-bewuste comparator.

8. **Float/variance-eenheid in de UI.** Blijft float in "werkdagen" (afgerond) of krijgt de gebruiker uur-precisie te zien? Dit raakt de weergave in TableEditor/TaskProperties/relatietabel én de variance-deltas (§1.5).
